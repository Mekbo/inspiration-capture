# 灵感捕捉器

将用户输入的灵感、想法、创意或待验证概念自动识别意图与内容类型，整理为完整结构化记录，保存到指定工作区的「灵感库」文件夹。

## 触发条件

当用户消息包含以下意图信号时触发：

- 输入一段灵感、创意、想法或头脑风暴内容
- 使用触发词：灵感、想法、创意、点子、记下来、备忘、头脑风暴、好主意、新想法、捕捉灵感、记录灵感、我要记个东西
- 描述一个"突然想到的"或"需要验证的"概念

用户可能随口说出，也可能直接要求"帮我记录一个灵感"。

## 工作流程

### 第一步：接收与确认

从用户消息中提取原始灵感内容。如果内容过于简略（仅几个词），主动追问补充以下信息：

1. 这个灵感具体是什么？（一句话概括）
2. 是什么触发的？
3. 想用在什么地方？

但不要机械地逐条问——用自然对话方式在 1-2 轮内补全。

### 第二步：意图识别与分类

根据 `references/record_template.md` 中的分类判定规则，自动判定灵感的 type（产品创意/内容创意/技术方案/文章灵感/商业模式/设计灵感/个人成长/问题与痛点/观察与洞见/其他）。

同时判定：
- **priority**（优先级）：
  - 高：用户明确说"很重要"或与当前核心工作/项目直接相关
  - 中：有价值但无紧迫性
  - 低：有趣的思考但不一定落地
- **source**（灵感来源）：阅读 / 对话 / 观察 / 梦境 / 散步 / 社交媒体 / 工作 / 其他
- **tags**：从内容中提取 3-8 个关键标签

### 第三步：结构化整理

按以下结构生成完整记录（模板详见 `references/record_template.md`）：

```yaml
---
title: "灵感标题（≤30字）"
type: "分类"
tags: ["标签1", "标签2", "标签3"]
priority: "高/中/低"
created: "ISO 8601 时间戳"
status: "灵感"
source: "灵感来源类型"
---
```

正文包含六个板块：
1. **灵感来源** — 触发背景
2. **核心内容** — 详细展开（问题/方案/价值/洞察）
3. **应用场景** — 落地可能
4. **待验证假设** — 需要验证的关键前提
5. **下一步行动** — 具体可执行步骤
6. **关联参考** — 相关链接/文章/人物

### 第四步：保存到灵感库

将记录保存到当前工作区的灵感库文件夹。工作区路径从系统 prompt 的 `<project_context>` 或 `<user_info>` 中的 `Workspace Folder` 获取（每次会话自动提供）。

保存路径构造规则：`{workspace}/灵感库/{YYYY-MM-DD}-{短标题}.md`

保存前：
1. 从当前会话上下文提取工作区根路径（`Workspace Folder`）
2. 拼接 `灵感库` 子目录，检查是否存在，不存在则创建
3. 检查是否有同名文件，如有则在文件名后追加 `-2`、`-3` 等序号

### 第五步：交付确认

保存完成后，向用户展示：
1. 文件保存路径
2. 关键结构化信息的摘要（标题、类型、优先级、标签）

不使用 `deliver_attachments` 或 `open_result_view`——灵感记录是知识库资产，不是交付物。

## 对话规范

- 第一优先：先理解用户说的内容，再结构化——不要机械套模板
- 如果用户说"帮我把这个灵感记下来"但没有给标题，从内容中提炼
- 如果用户在一次对话中连续记录多个灵感，每条独立保存
- type 分类让用户确认（用简要的一句），不强制但推荐："我把它归为「内容创意」类，对吗？"

## 扩展接口

灵感库支持后续扩展：
- 按类型/优先级/标签检索灵感
- 灵感→选题→内容创作的工作流串联
- 灵感成熟度追踪（灵感→待验证→进行中→已完成）

这些扩展功能在当前版本中不实现，但结构化存储格式已预留对应字段。
